# Non-Functional Requirements Checklist

当变更可能影响性能、稳定性、可观测性、兼容性、容量、灰度或回滚时使用本清单。

合并/发布结论统一按 `risk-assessment-template.md` 的 Merge Decision Rules 裁决。本文件只提供非功能专项审查项和高危样例。

## 性能与容量

- 数据库查询避免 N+1，并具备合适索引。
- 大列表使用分页、游标、流式处理、批处理或数量限制。
- 缓存行为包含失效策略、一致性预期、key 设计，必要时有击穿/雪崩保护。
- payload 大小、内存增长、CPU 密集任务、连接池、线程池和队列积压有边界。
- 同步请求延迟可解释，尤其是跨远程调用场景。

## 稳定性与韧性

- 远程调用和阻塞操作设置超时。
- 重试有上限，不会放大负载或重复副作用。
- 重复请求和消息具备幂等键、去重或安全状态流转。
- 关键依赖已考虑限流、熔断、降级和优雅退化。
- 部分失败时的用户体验和运维行为清晰。

## 可观测性

- 日志包含排障上下文，同时不泄露敏感数据。
- 错误码/错误信息可行动；客户端依赖时保持稳定。
- 关键成功、失败、延迟、积压状态有指标和告警。
- Trace ID 或 correlation ID 可跨服务边界和异步流程传递。
- 敏感操作、权限变更、数据导出和管理操作有审计日志。

## 兼容性、灰度与回滚

- API 变更保持向后兼容，或提供版本/迁移方案。
- 新旧代码可能并行运行时，数据库迁移具备前向兼容。
- 消息/事件 schema 变更兼容现有生产者和消费者。
- 发布风险高时使用 feature flag、灰度、配置默认值和 kill switch。
- 回滚方案覆盖代码、配置、数据库、缓存、生成客户端和外部契约。

## 非功能专项高危样例

以下问题通常应判为致命或高风险，并按 `risk-assessment-template.md` 的合并规则处理：
- 关键路径存在无边界查询、批处理、内存增长或队列积压。
- 发布关键流程中的远程调用链缺少超时或失败处理。
- 新增关键异步或跨服务流程没有可观测性。
- 破坏 API、数据库、消息或客户端兼容性，且没有迁移和回滚方案。
