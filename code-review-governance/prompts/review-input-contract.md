# Review Input Contract

把这份文件当作固定入口。先确定审查模式、审查对象和证据等级，再开始正式审查。

## 1. 先选审查模式

- 团队 PR 模式：有 PR、Reviewer、Owner、审批角色、CI 或发布流程。
- 个人开发 / 自审模式：没有 PR、没有审批角色、没有团队 owner 时默认使用。
- 证据不足预审：证据缺失时的降级模式，可与前两者叠加。

个人开发 / 自审模式下：

- `owner`、`批准角色`、`Tech Lead` 默认由当前开发者本人承担。
- `豁免` 退化为“自担风险记录”。
- `backlog/Issue` 可以是本地 TODO、仓库 issue、计划文档或审查输出里的待办段落。
- `合并/发布结论` 解释为“是否建议继续提交、继续扩大变更或进入发布”。

## 2. 固定审查入口

每次至少选择一个入口，按下面的最小输入开始。

### A. 审查本次 diff

最小输入：

- 变更范围：`当前工作区 / 某个 commit range / 某个 PR diff`
- 目标：这次改动想解决什么
- 重点文件或模块：可为空，但优先提供
- 已运行验证：测试命令、构建命令、手工验证，或明确写“未运行”
- 非范围项：哪些已知问题这次不处理

适用场景：

- 常规 feature、局部重构、小型修复、提交前自审。

### B. 审查某个模块

最小输入：

- 模块路径或模块名
- 审查目标：架构、可维护性、回归风险、边界清晰度，或专项维度
- 上下游：该模块依赖谁、被谁依赖；不知道也可留空
- 关键行为或主流程
- 现有验证：测试、运行方式、文档，或明确缺失

适用场景：

- 模块健康检查、重构前评估、复杂组件或服务治理。

### C. 审查整个仓库

最小输入：

- 仓库目标：项目做什么
- 优先关注范围：全仓 / 若干高风险区域 / 某一条关键链路
- 技术栈与运行边界：语言、框架、服务或应用数量；不知道可边读边补
- 可用证据：README、docs、ADR、测试、CI、部署文件
- 审查深度预期：快速扫描 / 深度架构审查

适用场景：

- 接手老项目、架构巡检、技术债盘点、发布前总审。

### D. 审查一次 bugfix 回归

最小输入：

- bug 描述：现象、期望行为、影响范围
- 修复范围：改了哪些文件、模块或流程
- 根因假设：已确认或待确认都可以
- 复现与验证：复现步骤、回归测试、手工验证，或明确缺失
- 相关历史：是否为同类问题反复出现

适用场景：

- 缺陷修复、线上事故修复、自测回归。

## 3. 噪音控制规则

默认优先审这些内容：

- 手写业务代码、配置、契约、迁移、测试和与当前风险直接相关的文档。

默认降权或按条件审这些内容：

- 生成代码：只在生成规则、生成结果异常、契约漂移、类型错误或依赖安全风险时深入审。
- 第三方目录、vendor、镜像产物：通常不做结构审查，只关注来源、版本、许可证、安全和是否被错误修改。
- 脚手架样板：除非已被项目长期承载业务逻辑，否则只做轻量一致性检查。
- 快照文件、golden 文件：重点看语义漂移和是否合理更新，不逐行挑格式问题。
- 迁移文件：重点看数据安全、兼容性、幂等性、回滚和顺序依赖，不对样式吹毛求疵。
- 临时实验代码：只有在会进入主路径、发布包或运行时可达时才按正式代码审；否则标记为待清理噪音。

每次审查都应说明：

- 本次明确纳入范围的区域。
- 本次明确排除或降权的区域。
- 排除理由是否会影响结论可信度。

## 4. 证据等级与降级策略

### 完整审查

适用条件：

- 需求/设计/范围清晰。
- 测试或运行验证证据基本可见。
- 风险可以被较完整地界定。

输出要求：

- 可以给正式合并/发布结论。

### 条件审查

适用条件：

- 范围清晰，但测试、兼容、迁移或运行时证据部分缺失。

输出要求：

- 可以给条件审查结论。
- 必须列出结论成立还缺哪些命令、场景或文档。
- 明确哪些问题是确认项，哪些只是高概率风险假设。

### 静态预审

适用条件：

- 只能做代码/结构静态阅读，无法确认关键行为、测试结果或运行上下文。

输出要求：

- 只下静态结构结论，不下完整发布结论。
- 明确写出“已确认的结构性问题”“待验证的风险假设”“无法判断的事项”。
- 至少补一个“下一步建议验证命令/场景”清单。

## 5. 默认输出附加项

无论是团队模式还是个人开发 / 自审模式，默认补上这三项：

- 先修的 1-3 个问题。
- 可以晚点修的事项。
- 应先补测试/验证，再决定是否改结构的问题。
