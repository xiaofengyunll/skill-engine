# Review Checklists

## 范围与噪音控制

- 先区分手写业务代码、生成代码、第三方目录、脚手架样板、快照、迁移文件和实验代码。
- 生成代码重点看生成规则、契约漂移、类型错误和不该出现的手工修改，不逐行挑风格。
- 第三方目录、vendor 和镜像产物默认不做结构审查，只看来源、版本、安全和是否被错误修改。
- 快照、golden 文件和大批量样板代码只检查语义漂移，不把主要注意力花在格式噪音上。
- 迁移文件优先审数据安全、兼容性、幂等性、顺序依赖和回滚。
- 临时实验代码如果不会进入主路径或发布包，可降权并标记待清理；如果运行时可达，就按正式代码审。

## 需求与设计一致性

- 核心需求是已实现、部分实现，还是缺失。
- 行为是否符合验收标准和已记录的非功能需求。
- 代码是否引入未记录的产品行为。
- 实现是否通过直接数据库/client 调用、隐藏捷径、重复流程或临时集成绕过设计。
- 临时 TODO/workaround 是否已经进入主路径。
- 测试是否覆盖文档化行为，而不只是覆盖实现细节。

## 架构与模块边界

- 每个模块是否只有一个内聚职责。
- 模块名和包名是否与业务或技术归属一致。
- 依赖是否单向且可理解。
- 是否存在循环依赖，或通过回调、事件处理、共享状态、共享库形成的隐式循环。
- 是否存在 god object、全能 manager、过载 Service、全局 store 或工具类倾倒区。
- 抽象是否用于稳定变化，而不是因为“应该套模式”才引入。

可操作判断：
- 模块边界混乱：一个模块承载多个业务域；模块访问其他模块私有表、私有 API 或内部类型；跨模块调用没有稳定契约；新增需求需要在多个无关模块同步改动。
- 模块边界局部模糊：职责大体清楚，但存在少量横向调用、重复 DTO、临时适配器或共享状态，短期可修复。
- 模块边界清晰：模块职责可用一句话说明；依赖方向稳定；外部只通过公开契约访问；数据和业务规则有明确 owner。
- Service 过重：单个 Service 处理鉴权、业务规则、事务、远程调用、持久化细节、DTO 转换、异常映射和日志策略中的多数职责。
- `shared/common/utils` 膨胀：承载业务规则、领域模型、跨上下文 DTO、临时兼容逻辑或需要多个业务模块同步发布的代码。

## 分层

后端：
- Controller 处理传输层关注点、校验编排和响应映射，而不是编写业务脚本。
- Service 编排业务用例和事务边界。
- Repository/Mapper 只处理持久化访问。
- 在项目架构支持时，领域逻辑应位于 domain service、entity 或 value object。
- 低层不依赖 web、controller 或传输 DTO。

前端：
- Page/view component 编排路由级流程。
- Component 负责 UI 渲染和局部交互。
- Hooks/composables 隔离可复用的有状态行为，而不是任意代码片段。
- Services/API clients 拥有请求细节。
- Stores 拥有共享状态，而不是承载所有业务过程。
- Utils 包含稳定纯 helper，而不是剩余业务逻辑。

## 函数设计

适合提取函数的情况：
- 提取后的名称表达业务概念或有意义的算法步骤。
- 能把副作用和纯逻辑分离。
- 能隔离可测试规则。
- 能移除有意义的重复。
- 能减少一个函数中的混合抽象层级。

避免提取函数的情况：
- 只制造一串单行包装。
- 调用方比内联逻辑更难读。
- 隐藏重要控制流或副作用。
- 把一个内聚序列拆成人为片段。
- 迫使读者为了局部理解在多个文件间频繁跳转。

审查项：
- 函数名、参数数量、参数形状、返回语义、null/empty/error 情况、副作用、幂等性和可测试性。

## 前端工程

- 巨型组件或页面同时包含 UI、请求、校验、转换和业务规则。
- 可复用 UI 组件内直接发 HTTP 请求。
- 可以通过组合、context 或 store 局部化的 props drilling。
- 使用全局 store 存放临时局部状态。
- Hooks/composables 混合无关关注点。
- API 层不一致、客户端重复、错误处理临时化。
- 类型/interface 重复、宽泛 `any`、DTO/view-model 分离不清。
- 样式分散、组件重复、魔法常量、design token 不一致。

## Java/Spring 工程

- Controller 过重，或 Service 过于脚本化。
- 事务边界过高、过低，或跨远程调用但没有清晰一致性设计。
- DTO/VO/DO/Entity 转换缺失、重复或不一致。
- Mapper/Repository 包含业务决策。
- Service-to-service 调用造成隐藏循环或归属不清。
- 领域逻辑分散在 controller、service、mapper、scheduled job 和 listener 中。
- 包结构混合技术关注点和业务关注点，且规则不一致。
- 配置重复、魔法值、缺少枚举/常量、滥用工具类。
- 异常处理和日志缺少可行动上下文，或泄露内部信息。

高风险样例：
- 事务包裹远程调用，但没有超时、补偿或幂等设计。
- Repository/Mapper 决定业务状态流转。
- Controller 直接写数据库，或调用多个 Repository 实现业务流程。
- Public DTO 暴露持久化内部结构，或变更时没有兼容处理。
- Security context、tenant ID 或 user scope 在服务间读取不一致。

## Python/FastAPI 工程

- router、service、repository、schema 之间职责是否清晰。
- Pydantic schema、ORM model、领域模型是否混用失控。
- 依赖注入、request scope、DB session 生命周期是否清晰且可预测。
- background task、async 调用、外部 client 和异常映射是否有明确边界。
- API 契约、错误响应、分页/过滤/排序语义是否稳定。

高风险样例：
- request/session 生命周期泄漏，导致事务边界或连接管理不可靠。
- router 直接承载大量业务逻辑，service 退化为薄包装。
- schema/model 复用不当，导致对外契约暴露内部存储结构。

## Node/Nest/Express 工程

- controller、service、module、middleware、guard、interceptor 的职责是否稳定。
- DTO/schema 校验、序列化、错误处理和日志是否一致。
- 异步错误、未处理 promise、资源释放和超时语义是否明确。
- shared utils、base service、common module 是否开始承载业务规则。

高风险样例：
- controller 或 middleware 直接处理核心业务和持久化。
- guard/interceptor 混入业务规则，导致权限与业务耦合。
- 多个模块复制 DTO、校验规则或 client 封装，后续容易漂移。

## Go / CLI / 脚本工程

- package 是否围绕真实边界组织，而不是按技术碎片随意堆放。
- interface 是否用于稳定替换点，而不是为了 mock 到处抽象。
- context、cancel、timeout、goroutine 生命周期是否正确传播。
- 命令入口、副作用、配置来源、退出码和日志是否适合自动化运行。
- 脚本是否幂等，重跑是否安全，失败后是否容易恢复。

高风险样例：
- context 丢失或 goroutine 无法收敛，造成资源泄漏。
- CLI/脚本在失败时留下半完成状态，但没有回滚或重入设计。
- package 之间通过全局状态、共享可变配置或隐式副作用耦合。

## 长期维护风险

- 某个模块会变成无关未来变更的默认放置位置。
- 某个服务拥有数据或行为，导致另一个服务必须持续协调。
- 某种设计让新增变体需要修改许多远处文件。
- 局部 workaround 变成公开契约的一部分。
- 测试缺口让重构不安全。
- 重复模式足够相似以至于令人困惑，但又尚未稳定到适合抽象。

## 团队协作

- 新成员能否快速定位负责模块。
- 两名工程师能否修改相邻功能，而不产生频繁冲突或归属混乱。
- API 和 DTO 是否从调用方视角命名，并足够稳定可复用。
- 日志、异常和注释是否有助于事故响应。
- 约定是否能从现有代码中看出，是否有文档并被一致遵守。

## 安全清单

- 认证：endpoint、job、callback、admin tool 和内部 API 需要正确身份检查。
- 授权：对象级权限、角色检查、租户检查和数据范围在可靠位置执行。
- 输入校验：request body、query param、header、file、callback、message 和导入数据已校验。
- 注入：SQL、NoSQL、command、template、path traversal、LDAP、expression language 和 SSRF 风险已控制。
- Web 安全：XSS、CSRF、CORS、cookie flag、redirect URL 和内容渲染安全。
- 密钥：key、token、password、certificate 和 connection string 不会被提交、记录或返回。
- 敏感数据：日志、错误、导出、指标和审计记录会脱敏或省略敏感信息。
- 依赖：新增或升级依赖的漏洞、许可证和供应链风险可接受。
- 文件处理：上传类型、大小、存储路径、病毒扫描、预览/渲染和下载权限已控制。
- 反序列化和回调：签名、时间戳、防重放、白名单和 parser 安全已检查。
- 多租户数据：tenant isolation 在查询、缓存、索引、队列和后台任务中执行。

## 测试与验证清单

- 单元测试覆盖业务规则、校验器、边界条件、数据转换和确定性分支。
- 集成测试覆盖数据库查询、事务、缓存行为、消息、框架装配和远程客户端。
- 契约测试覆盖 public API 字段、错误码、事件 schema、callback、生成客户端和向后兼容。
- 端到端或流程测试覆盖关键用户旅程和跨服务流程。
- Bug fix 增加回归测试；无法自动化时记录人工回归场景。
- schema/data migration 有迁移测试或演练证据。
- 并发和幂等测试覆盖重复提交、异步任务、重试、分布式锁和批处理。
- 验证命令应被准确记录，包括失败或跳过的命令及原因。

风险分类：
- 高风险测试缺口：核心流程、安全、数据迁移、事务/并发、异步处理、兼容性或回滚缺少测试。
- 中风险测试缺口：局部业务规则、边界条件、错误分支、DTO 转换或 UI 交互缺少测试。
- 建议优化测试缺口：主路径已覆盖，剩余测试主要改善边界可读性或低风险变体。

## 非功能质量清单

- 性能：分页、批处理、查询次数、索引使用、缓存行为、内存增长、payload 大小和同步延迟。
- 稳定性：超时、重试、幂等、限流、熔断、降级、重复消息处理和部分失败行为。
- 可观测性：结构化日志、错误码、指标、trace ID、correlation ID、审计记录和可告警信号。
- 兼容性：API versioning、字段语义、错误码、数据库迁移兼容、客户端兼容、配置默认值和生成客户端。
- 容量：队列积压、批量大小、连接池、线程池、调度频率和流量峰值行为。
- 发布安全：feature flag、灰度发布、回滚路径、数据回滚、配置回滚和发布期间监控。
