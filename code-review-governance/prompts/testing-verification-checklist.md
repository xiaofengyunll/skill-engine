# 测试与验证清单

使用本清单判断审查是否有足够证据支持合并、发布或豁免结论。

## 测试类型

- 单元测试：确定性业务规则、校验器、数据转换、边界条件、错误映射和纯函数。
- 集成测试：数据库、事务、缓存、框架装配、远程客户端、消息、调度器和持久化映射。
- 契约测试：public API 字段、错误码、事件 schema、callback payload、生成客户端、RPC/proto 兼容和共享 DTO。
- 端到端测试：关键用户旅程、跨服务流程、支付/订单/认证/数据写入路径和发布冒烟检查。
- 回归测试：每个 bug fix 都应包含自动化回归测试；如果不现实，则记录人工验证场景。

## 专项验证要求

- 数据迁移：验证正向迁移、必要时验证与旧代码兼容、回滚或修复方案，以及生产数据假设。
- 事务/并发：验证重复提交、部分失败、锁行为、重试、幂等和重复消息。
- 异步任务：验证调度、重试、poison message、恢复、去重、顺序假设和可观测性。
- API 兼容：验证旧客户端、可选/新增字段、默认值、错误码、分页、排序和版本协商。
- 安全：验证权限拒绝、对象级访问、租户隔离、输入校验和敏感数据脱敏。

## 覆盖率说明

覆盖率是信号，不是裁决规则。不要因为覆盖率高就批准高风险代码，也不要因为全局覆盖率不完美就阻塞低风险变更。优先使用与变更行为直接相关的场景证据。

- 人工验证：
- 剩余未验证风险：

## 风险分类

- 高风险：核心流程、安全、数据迁移、事务/并发、异步处理、兼容性或回滚缺少证据。
- 中风险：局部业务规则、边界条件、错误分支、DTO 转换或 UI 交互缺少证据。
- 建议优化：主路径已覆盖，剩余测试主要改善清晰度或低风险边界。
