# Review Workflow Reference

## 推荐工作流

1. 准入检查与证据收集
   - 识别仓库类型、语言栈、构建系统、模块布局、运行边界和本次变更范围。
   - 收集需求文档、设计文档、任务计划、API 契约、ADR、README、包/构建清单、配置文件和测试。
   - 如果用户要求审查 PR 或变更集，应把变更文件放回周边架构中审查，而不只看 diff。
   - 记录 CI/build/lint/test 状态是否可见。
   - 将缺失的需求、设计、测试、迁移、兼容或回滚证据记录为审查缺口。
   - 如果准入证据不完整，按预审处理，并明确哪些结论是有条件的。

2. 架构图谱
   - 梳理模块、包根、前端应用区域、后端服务、共享库、数据存储、消息队列、RPC 客户端和外部集成。
   - 识别依赖方向和所有权边界。
   - 标记不清晰或可疑区域，作为后续深入审查对象。

3. 一致性检查
   - 对照需求、设计和任务计划检查当前实现。
   - 标记核心流程缺失、未记录的范围扩张、设计漂移、架构绕过、临时实现长期化，以及 API 或数据契约不匹配。

4. 架构检查
   - 检查模块职责、分层职责、跨层访问、耦合、循环依赖、数据归属、共享代码和抽象层级。
   - 区分健康的务实简化和真实的设计侵蚀。

5. 可维护性检查
   - 检查文件职责、函数拆分、命名、类型/模型、重复逻辑、副作用、可测试性和局部可读性。
   - 同时识别结构不足和过度抽象。

6. 领域专项检查
   - 前端：组件、hooks/composables、stores、services/API layer、路由、类型定义、样式、常量、可复用 UI、状态归属。
   - Java/Spring：Controller、Service、Repository/Mapper、事务边界、DTO/VO/DO/Entity、领域逻辑位置、包结构、配置、异常、日志。
   - 微服务：服务边界、数据归属、RPC 链路、网关职责、共享模块、重复业务规则、契约版本。

7. 安全、测试与非功能检查
   - 安全：认证、授权、对象权限、租户隔离、输入校验、注入、XSS/CSRF、密钥、敏感日志、依赖、上传、回调和反序列化。
   - 测试：单元、集成、契约、端到端、回归、迁移、并发、异步任务、回滚和人工验证证据。
   - 非功能：性能、容量、超时、重试、幂等、限流、降级、可观测性、兼容性、灰度和回滚。

8. 风险分级报告
   - 按严重等级排列发现，优先列出风险最高的问题。
   - 说明每个问题为什么重要，以及长期影响是什么。
   - 同时给出立即修复、架构演进和治理动作。
   - 按 `risk-assessment-template.md` 的 Merge Decision Rules 给出合并/发布结论：`可以合并`、`暂缓合并`、`阻塞合并` 或 `可带债发布`。
   - 记录阻塞问题、必要修复、已批准豁免、技术债跟踪、验证命令和未验证风险。

9. 用户确认后再生成后续产物
   - 除非用户要求后续文档，否则在审查摘要和问题列表后停止。
   - 询问是否需要继续生成完整审查报告、修复计划、架构治理建议、模块重构方案、技术债路线或分阶段整改计划。
   - 只生成用户选择的产物，并确保内容基于本次审查证据。

## 团队 Review 流程

当审查属于团队 PR 流程时使用：

1. 作者自检
   - 使用 `pr-description-template.md` 说明范围、风险、测试证据、迁移/回滚说明和已知权衡。
   - 涉及时链接需求、Issue、设计、API、schema 或 ADR 证据。

2. 首轮审查
   - 普通 Reviewer 检查局部正确性、可维护性、测试和协作约定。
   - 架构 Reviewer 在变更跨边界时检查模块归属、分层、契约、数据归属和演进风险。
   - 安全 Reviewer 在涉及权限、输入、密钥、依赖、租户、文件、回调或数据暴露时做专项审查。
   - QA/Test Owner 检查关键流程的验收覆盖、回归范围和验证充分性。

3. 修改与回应
   - 作者修复问题、解释问题不适用的原因，或申请豁免。
   - Reviewer 只复查受影响区域和新引入的风险。

4. 批准与合并结论
   - 只有满足准出条件后才能合并。
   - 阻塞问题必须在批准前解决。
   - 被豁免的问题必须记录 owner、截止时间、批准角色和 backlog。
   - 高风险豁免必须按 `risk-assessment-template.md` 的 Approval Matrix 执行；涉及安全、数据、合规、发布、回滚或核心链路时，必须追加对应专项 Reviewer 或发布负责人。

5. 合并后跟踪
   - 跟踪技术债、后续缺陷、监控任务，以及回滚/灰度观察项。
   - 只有修复完成或约定的验证证据存在后，才能关闭审查发现。

## 角色职责

| 角色 | 职责 | 必要证据 |
| --- | --- | --- |
| 作者 | 说明变更范围、风险、测试、兼容性、迁移和回滚。 | PR 描述、关联 issue/design、测试命令、已知缺口。 |
| 普通 Reviewer | 检查正确性、可维护性、局部设计、测试、命名、异常、日志和约定。 | 带文件/行号证据和风险等级的问题。 |
| 架构 Reviewer | 检查模块边界、归属、分层、契约、数据归属、耦合和可逆性。 | 架构发现、必须修复项或豁免建议。 |
| 安全 Reviewer | 检查权限、租户隔离、注入、密钥、依赖、日志、上传、回调和数据暴露风险。 | 安全清单结果，以及安全类阻塞/豁免判断。 |
| QA/Test Owner | 检查验收覆盖、回归范围、人工验证和发布验证。 | 测试证据、未验证风险、发布验证建议。 |
| 模块 Owner/Tech Lead | 按 `risk-assessment-template.md` 的 Approval Matrix 参与普通高风险、专项高风险和发布级风险批准；涉及安全、数据、合规、发布、回滚或核心链路时，必须追加对应专项 Reviewer 或发布负责人。 | 批准记录、owner、截止时间、backlog 链接；专项或发布级风险需包含对应批准角色。 |

## SLA 与争议处理

- 高优先级 PR 首次响应：尽量当天；否则 24 小时内。
- 普通 PR 首次响应：1-2 个工作日内。
- 安全、数据或发布阻塞问题：立即升级给 Tech Lead 或责任 Owner。
- Reviewer 意见不一致时，优先按文档化证据判断，再由模块 Owner 或架构 Reviewer 裁决。
- 如果结论接受风险，必须记录为豁免或技术债，不要只留在评论中。

## 推荐审查顺序

实际仓库审查按以下顺序：

1. 阅读顶层文件：README、构建清单、包清单、模块描述、部署文件和文档。
2. 搜索文档：`docs`、`requirements`、`design`、`planning`、`adr`、`architecture`、`api`、`openapi`、`proto`。
3. 列出模块和高层包结构。
4. 检查变更文件或目标模块。
5. 检查相邻模块，理解本地约定。
6. 检查目标行为相关的测试和契约。
7. 检查变更涉及的安全、测试和非功能维度。
8. 尽量用具体文件/行号写出发现。
9. 区分已确认问题、假设和审查缺口。
10. 给出合并/发布结论和必要后续动作。
11. 询问用户是否需要后续产物，不默认生成大文档。

## 审查优先级策略

按以下顺序审查：

1. 发布阻塞问题，以及需求/设计不一致。
2. 后续难以逆转的架构违规。
3. 模块/服务边界和数据归属问题。
4. 事务、一致性、安全、可观测性和错误处理风险。
5. 可维护性和协作风险。
6. 重复、命名、风格和局部优化。

核心架构风险未解决前，不要把主要注意力放在格式问题上。

## 大型项目审查策略

对于大型仓库：

1. 深入阅读前先建立模块清单。
2. 识别关键路径：认证、订单/支付、数据写入、异步任务、共享库、网关、部署配置、生成客户端和公开 API。
3. 跨层抽样代表性模块，不要一开始就逐文件阅读。
4. 围绕风险信号扩展：大文件、循环依赖、异常层级直接访问 DB/client、重复 DTO、不一致 API client、测试缺失。
5. 分阶段输出审查：
   - 立即发布风险。
   - 架构一致性风险。
   - 技术债和治理 backlog。
   - 建议的下一轮审计范围。
6. 询问用户下一步需要哪种详细产物，不自动把每个阶段扩写成独立文档。

## 多模块工程策略

检查：

- 每个模块是否有清晰 owner 和职责。
- 依赖方向是否符合预期分层。
- common/shared 模块是否承载稳定抽象，而不是成为业务逻辑倾倒区。
- DTO/API/model 包是否重复或分叉。
- 构建配置和依赖版本是否适当集中管理。
- 模块边界是否支持独立变更，而不是制造流程负担。

风险信号：

- `common`、`core`、`base` 或 `utils` 比业务模块增长更快。
- Feature 模块之间横向依赖，且没有清晰契约。
- 共享模块包含多个限界上下文的领域代码。
- 测试需要启动无关模块。

## 微服务审查策略

检查：

- 服务边界：一个服务拥有一个内聚的业务能力。
- 数据归属：服务不写入其他服务的私有表。
- API 边界：外部契约稳定，必要时版本化，且不被内部持久化细节塑形。
- RPC 链路：避免单次用户请求形成过长同步链路；如果存在，必须明确管理延迟和失败行为。
- 共享代码：避免 common 模块耦合服务发布节奏或泄露领域逻辑。
- 网关：负责路由、认证、限流和轻量聚合；不要把业务逻辑移入网关。
- 可观测性：跨服务流程具备日志、指标、trace ID 和失败语义。

按可逆性判断微服务问题严重程度。错误的数据归属和共享领域库耦合，通常比局部代码风格问题更严重。
