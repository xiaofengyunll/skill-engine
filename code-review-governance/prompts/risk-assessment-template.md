# Risk Assessment Template

本文件是合并/发布结论与阻塞规则的唯一权威来源。其他模板可以引用本文件，或列出安全、测试、非功能等专项风险样例，但不得重新定义合并决策口径。

## Severity Definitions

### 致命

适用场景：
- 破坏核心需求或验收标准。
- 造成严重数据正确性、安全、事务或发布失败风险。
- 违反下游模块依赖的关键架构契约。
- 不立即修复就无法安全发布。

默认合并结论：阻塞合并。

示例：
- 核心流程不可用或与验收标准相反。
- 越权访问、租户数据泄露、密钥暴露、注入风险或敏感日志泄露。
- 数据迁移会破坏数据，或无法安全回滚。
- 对外 API、数据库 schema 或消息契约破坏已知客户端，且没有迁移或兼容方案。

### 高风险

适用场景：
- 造成明显的长期耦合、服务边界、数据归属或维护性风险。
- 使后续变更显著变贵或更容易引入缺陷。
- 引入事务、一致性、可观测性或错误处理的不确定行为。
- 造成后续难以逆转的架构漂移。

默认合并结论：
- 高风险已修复，且必要验证完成：可以合并。
- 高风险未修复，但已有批准角色、owner、截止时间和 backlog：可带债发布。
- 高风险未修复且未豁免：暂缓合并或阻塞合并。
- 数据、安全、兼容、核心链路类高风险：默认阻塞合并；只有发布级批准和完整风险记录后才可按豁免流程处理。
- 若同一问题同时命中多条规则，按更严格的合并结论执行。数据、安全、兼容、核心链路类高风险必须满足发布级批准，不适用普通高风险豁免口径。

示例：
- 服务边界或数据归属被绕过，且后续难以回收。
- 关键路径缺少回归、集成、契约或迁移验证。
- 长同步 RPC 链路缺少超时、重试、降级或可观测性设计。
- shared/common 模块承载多个业务上下文的业务规则。

### 中风险

适用场景：
- 问题局部存在，但会明显影响可读性、可测试性、扩展性或协作效率。
- 产生可能继续增长的重复逻辑或职责不清。
- 应尽快修复，但通常不单独阻塞发布。

默认合并结论：评审者确认后可以合并，必要时作为近期技术债跟踪。

示例：
- 局部重复会让后续变体更容易出错。
- 边界条件或错误分支缺少测试，但核心路径已验证。
- 组件或 Service 混合了相关职责，但尚未突破模块边界。

### 建议优化

适用场景：
- 风险有限，但调整后会更清晰。
- 改善命名、注释、测试可读性、常量、局部结构或局部一致性。
- 不应夸大成架构问题。

默认合并结论：不阻塞合并。

示例：
- 命名、注释、常量提取或局部测试可读性可以改善。
- 存在少量重复，但抽象尚未稳定。

## Risk Scoring

每个问题按 1-5 分评估：

- Impact：失败后果或未来成本有多严重。
- Probability：造成缺陷或交付摩擦的概率。
- Reversibility：后续修复或回滚有多困难。
- Spread：影响多少模块、服务或团队。
- Evidence：证据是否具体。

建议映射：

- 20-25: 致命或高风险，取决于发布影响。
- 15-19: 高风险.
- 9-14: 中风险.
- 5-8: 建议优化.

不要只按分数裁决，必须说明工程理由。

## Merge Decision Rules

规则优先级：若同一问题同时命中多条规则，按更严格的合并/发布结论执行。数据、安全、兼容、核心链路类高风险优先于普通高风险规则，必须满足发布级批准，不适用普通高风险豁免口径。

| 条件 | 合并/发布结论 |
| --- | --- |
| 存在任何致命问题 | 阻塞合并 |
| 数据、安全、兼容、核心链路类高风险未修复 | 阻塞合并，除非有发布级批准和完整豁免记录 |
| 高风险已修复，且必要验证完成 | 可以合并 |
| 高风险未修复，但已有批准角色、owner、截止时间和 backlog | 可带债发布 |
| 高风险未修复且未豁免 | 暂缓合并；若影响数据、安全、兼容或核心链路，则阻塞合并 |
| 只有中风险或建议优化，且验证充分 | 可以合并 |
| 关键验证证据缺失且风险无法界定 | 暂缓合并或阻塞合并 |

个人开发 / 自审模式说明：

- 继续沿用上表，但把“合并/发布结论”解释为“当前是否建议继续提交、继续扩大变更或进入发布”。
- 证据不足时，最多给“条件审查”或“静态预审”结论，不要假装完成了正式发布判断。
- 数据、安全、兼容、核心链路类高风险在自审模式下仍默认按阻塞处理；如果开发者决定暂时接受，必须记录自担理由、截止时间、回滚方案和待补验证。

## Approval Matrix

| 风险类型 | 最低批准要求 |
| --- | --- |
| 普通高风险 | 模块 Owner 或 Tech Lead |
| 安全/数据/合规高风险 | 对应专项 Reviewer + Tech Lead |
| 发布/回滚/核心链路高风险 | 发布负责人 + Tech Lead |
| 跨团队接口高风险 | 接口 Owner 或调用方代表参与确认；必要时 Tech Lead 共同批准 |
| 致命问题 | 原则上不接受豁免；如需例外，必须由 Tech Lead、发布负责人和相关专项 Owner 共同批准并记录发布级风险 |

发布级批准指对线上发布风险承担明确责任的批准记录，至少包含批准角色、批准理由、影响范围、回滚方案、监控/验证方式、有效期和关闭条件。

个人开发 / 自审模式下，若没有独立批准角色，可用“自担风险记录”代替批准记录，但字段仍应尽量完整，至少包含：接受理由、影响范围、最晚处理时间、待补验证、回滚或兜底方式。
