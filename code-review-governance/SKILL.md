# Code Review Governance

从技术负责人、架构师和工程治理审查者的视角审查代码库。重点关注需求/设计一致性、架构一致性、模块职责、可维护性、可扩展性，以及协作或自维护风险，不要把审查降级成格式、lint 或孤立 bug 检查。

## Core Principle

优先追求工程合理性，而不是教条。只有当抽象、拆分、设计模式、微服务、函数式写法或新增层级能降低真实复杂度、澄清归属、提升变更安全性，或符合既有架构时，才建议引入。

## Review Mode Selection

先选模式，再套用后续模板：

- 团队 PR 模式：存在 PR、Reviewer、Owner、审批角色、CI 或发布流程时使用完整团队治理口径。
- 个人开发 / 自审模式：没有 PR、没有审批角色、没有团队 owner 时默认使用。仍沿用同一套风险等级，但把“合并/发布结论”解释为“当前变更是否适合继续提交、继续迭代或进入发布”。
- 证据不足预审：无论团队还是个人，只要需求、设计、测试或运行验证证据缺失，就降级为条件审查或静态预审，不假装给出完整发布结论。

个人开发 / 自审模式下使用这些退化规则：

- `模块 Owner`、`Tech Lead`、`批准角色` 默认退化为当前开发者本人。
- `豁免` 退化为显式的“自担风险记录”，至少写明为什么暂不修、最晚何时回头处理、下一次触发复审的条件。
- `backlog/Issue` 可以是本地 TODO、仓库 issue、计划文档或审查结果中的待办段落，不要求必须有团队系统。
- `阻塞合并 / 暂缓合并 / 可带债发布 / 可以合并` 分别解释为：先不要继续提交或发布 / 暂缓继续扩大变更 / 可以带债继续但必须记账 / 当前可以继续。

## Review Governance Baseline

把审查结论写成可执行判断，而不只是“建议”。每次 review 都要明确：
- 是否满足审查准入条件。
- 是否存在必须阻塞合并的问题。
- 哪些问题可以带债发布，以及豁免责任人或自担责任记录、有效期和跟踪方式。
- 已验证了什么，哪些风险尚未验证。

### Review Entry Criteria

开始正式审查前先确认准入条件。条件不足时仍可做预审，但必须在结论中标记为“证据不足”。固定入口和最小输入模板见 `prompts/review-input-contract.md`。

- PR、diff、模块、仓库或 bugfix 回归范围清晰：至少能说清审查对象、目标和不在范围内的事项。
- CI/build/lint/test 的当前状态可见；若未运行，必须说明原因。
- 涉及 API、数据库、消息、权限、配置、部署或前端交互变更时，有对应契约、迁移、兼容或验证说明。
- 涉及架构边界、模块职责或跨团队接口时，有设计说明、ADR、接口文档或现有约定可对照。
- Bug fix 应提供复现说明或回归测试证据；没有证据时应作为测试缺口记录。

### Review Exit Criteria

合并/发布结论以 `prompts/risk-assessment-template.md` 的 Merge Decision Rules 为唯一权威来源。团队模式按合并/发布解释，个人开发 / 自审模式按“是否建议继续提交或发布”解释。这里保留简要口径：

- 高风险已修复，且必要验证完成：可以合并。
- 高风险未修复，但已有批准角色、owner、截止时间和 backlog：可带债发布。
- 高风险未修复且未豁免：暂缓合并或阻塞合并。
- 数据、安全、兼容、核心链路类高风险：默认阻塞合并；只有发布级批准和完整风险记录后才可按豁免流程处理。
- 若同一问题同时命中多条规则，按更严格的合并结论执行；专项高风险不适用普通高风险豁免口径。
- 只有中风险或建议优化，且验证充分：可以合并，必要时进入技术债 backlog。
- 证据不足时，最多只给“静态预审”或“条件审查”，并明确还缺哪些命令、场景或文档。

### Blocking Rules

阻塞规则不要在多个文档中重复定义。统一读取 `prompts/risk-assessment-template.md` 的 Merge Decision Rules；安全、测试、非功能清单只提供专项风险样例，不单独决定合并结论。

### Waiver And Technical Debt Tracking

允许豁免时必须记录：

- 豁免问题、风险等级、影响范围和不立即修复的理由。
- 批准角色：必须按 `prompts/risk-assessment-template.md` 的 Approval Matrix 执行；普通高风险由模块 Owner 或 Tech Lead 批准，安全/数据/合规、发布/回滚/核心链路、跨团队接口和致命问题例外按对应专项批准要求执行。个人开发 / 自审模式下默认由当前开发者本人承担，但仍要记录理由、截止时间、验证缺口和回滚思路。
- 有效期：默认不超过一个迭代；超过一个迭代必须重新评估。
- 技术债条目：包含 owner、优先级、截止时间、验证方式和关闭条件。
- 复审机制：到期未关闭时升级到治理 backlog 或下一轮架构评审。

## Review Workflow

1. 确定审查范围。
   - 优先使用 `prompts/review-input-contract.md` 中的固定入口：审查本次 diff、审查某个模块、审查整个仓库、审查一次 bugfix 回归。
   - 识别项目类型：前端、Java/Spring、全栈、多模块、微服务、库，或混合类型。
   - 查找需求文档、设计文档、任务计划、ADR、README、API 契约、模块文档、包清单、构建文件和测试。
   - 先做噪音控制：区分 authored source、生成代码、第三方目录、脚手架样板、快照、迁移和实验代码，避免把主要精力浪费在低价值区域。
   - 如果文档缺失，明确说明，并基于可观察的架构和命名约定进行审查。

2. 建立架构图谱。
   - 总结模块、分层、服务边界、入口点、共享包、依赖方向、数据归属和集成路径。
   - 对照文档中的预期架构和代码中的实际结构。
   - 对大型仓库，先抽样代表性模块，再扩展到高风险或已变更区域。

3. 先审查一致性，再看细节。
   - 检查实现是否满足已记录的需求、设计和任务计划。
   - 标记核心需求缺失、自行扩展范围、架构绕过、临时实现长期化和设计漂移。

4. 审查架构与可维护性。
   - 检查模块边界、分层、耦合、循环风险、职责集中、重复逻辑和长期变更成本。
   - 判断复杂度是必要复杂度、偶发复杂度、设计不足，还是过度设计。

5. 审查框架相关工程实践。
   - 前端：检查组件边界、hooks/composables、services/API layer、stores、状态归属、类型、样式和可复用 UI。
   - Java/Spring：检查 Controller/Service/Repository 分离、事务边界、DTO/VO/DO/Entity 使用、领域逻辑位置、包结构、配置、异常处理和日志。
   - Python/FastAPI：检查 router/service/repository 分层、Pydantic schema 边界、依赖注入、事务/session 生命周期、后台任务和异常映射。
   - Node/Nest/Express：检查 controller/service/module 中的职责、DTO/schema 校验、中间件/guard/interceptor 归属、异步错误处理和共享 util 膨胀。
   - Go：检查 package 边界、interface 使用是否必要、context 传递、错误处理、并发/取消语义和基础设施耦合。
   - CLI/脚本仓库：检查命令入口、参数契约、副作用隔离、配置来源、幂等性、回滚/重跑安全和日志输出。
   - 微服务：检查服务归属、数据归属、API 边界、共享模块、RPC 链路、网关职责和重复业务规则。

6. 输出按风险排序的审查结论。
   - 按严重等级和架构影响排序，优先列出问题。
   - 每个问题包含位置、原因、为什么重要、长期影响、建议方案，以及是否建议立即修改。
   - 不只给代码级修复，也要给出架构层建议和治理动作。

7. 给出合并/发布结论。
   - 标记 `可以合并`、`暂缓合并`、`阻塞合并` 或 `可带债发布`。
   - 记录阻塞项、必须修复项、已批准豁免、验证命令和未验证风险。
   - 证据不足时，显式降级为 `静态预审` 或 `条件审查`，不要假装完成了发布判断。

8. 生成后续产物前先询问。
   - 审查后先输出简洁的审查摘要。
   - 除非用户要求，不自动创建大型文档、修复计划、重构计划或治理路线。
   - 提供可选后续产物，让用户选择下一步生成内容。

## Required Review Dimensions

### 需求与设计一致性

检查实现是否：
- 偏离需求文档、设计文档或任务计划。
- 在开发过程中增加未记录的范围。
- 遗漏核心需求或验收标准。
- 实现了与设计相矛盾的行为。
- 绕过既定架构、API 契约或模块归属。
- 让临时方案或 workaround 进入长期主路径。

### Architecture Reasonableness

检查模块划分：
- 模块职责和归属是否清晰。
- 模块边界和依赖方向是否合理。
- 是否存在循环依赖、强耦合、重复业务逻辑、大而全模块、工具类倾倒区和 god object。

使用具体判断样例：
- 模块边界混乱：一个模块同时承载多个业务域；上层模块直接访问其他模块私有数据； feature 模块横向互调且无稳定契约；修改一个需求需要跨多个无关模块同步改动。
- Service 过重：单个 Service 同时处理鉴权、编排、业务规则、持久化细节、远程调用、DTO 转换和异常映射；新增分支只能继续堆进同一个方法或类。
- shared/common 膨胀：包含具体业务规则、领域 DTO、跨上下文工具、临时适配器；被多个模块依赖后迫使它们同版本发布。
- API 兼容风险：删除/重命名字段、改变字段语义、改变错误码、改变分页/排序默认行为、收紧权限、改变幂等键或回调签名，都必须做兼容性检查。

检查分层架构：
- Java/Spring 中 Controller/Service/Repository 职责是否清晰。
- 前端项目中 Page/component/hooks/services/stores/utils 职责是否清晰。
- 是否存在跨层调用、直接访问底层能力、绕过 service 抽象等问题。

检查微服务治理：
- 服务职责和边界是否清晰。
- 数据归属是否正确。
- 服务之间是否重复业务逻辑。
- 共享模块是否膨胀。
- RPC 链路是否不合理，或网关职责是否泄漏。

### Code Structure And Maintainability

检查文件职责：
- 超大文件、复杂页面、过载 Service、过大的 store/hooks、泛化 utils、DTO/VO/Entity 混乱，以及业务关注点混杂。

检查函数设计：
- 不要只按函数长度判断。
- 当提取函数能命名真实业务概念、隔离副作用、提升可测试性、移除有意义重复或分离抽象层级时，优先提取。
- 当提取只会产生一行包装、过多同层嵌套、深跳转链、人为间接层或“为了封装而封装”时，避免提取。
- 审查命名、参数、返回值、副作用、可测试性、重复和调用链可读性。

### Collaboration And Governance

检查命名、注释、目录规则、提交、模块约定、API 契约、异常、日志、常量、魔法值、枚举、配置分层和理解成本。判断实现是否适合多人协作，或在个人项目中保持长期自维护清晰度。

### Security

当代码涉及身份、权限、用户输入、持久化、文件、Web 视图、回调、依赖更新、密钥、日志或多租户数据时，将安全作为独立维度审查。

检查认证、授权、对象级权限、租户隔离、输入校验、SQL/命令注入、XSS/CSRF、反序列化、文件上传、webhook/callback 校验、依赖漏洞、密钥处理、敏感日志脱敏和安全默认值。

### Testing And Verification

要求提供与变更相匹配的证据：
- 单元测试覆盖确定性业务规则、数据转换、校验器和边界条件。
- 集成测试覆盖数据库、缓存、事务、消息、远程客户端和框架装配。
- 契约测试或 schema 检查覆盖公开 API、RPC、事件、回调、生成客户端和共享 DTO。
- 端到端或流程测试覆盖关键用户旅程和跨服务流程。
- Bug fix 应补充回归测试；如果确实不适合自动化，应记录人工验证命令或场景。

测试缺口风险判断：
- 高风险：核心链路、权限、数据迁移、并发/事务、异步任务、兼容性或回滚逻辑缺少验证。
- 中风险：局部业务规则、边界条件、错误分支、DTO 转换或 UI 交互缺少验证。
- 建议优化：已有主路径验证，但可补充少量边界或可读性更好的测试。

### Non-Functional Quality

检查性能、稳定性、可观测性、兼容性、容量、灰度和回滚：
- 性能：N+1 查询、缺失分页、无边界循环、缓存失效、批量大小、内存增长和同步延迟。
- 可靠性：超时、重试、幂等、限流、熔断、降级、重复消息处理和部分失败行为。
- 可观测性：可行动日志、错误码、指标、trace/span ID、correlation ID、审计日志和可告警信号。
- 兼容性：API 版本、数据库迁移兼容、客户端兼容、feature flag、灰度发布、回滚路径和配置默认值。

## Severity Model

使用以下等级：
- 致命：破坏核心需求，造成数据、安全、正确性或发布失败，违反关键架构契约，或导致无法安全发布。
- 高风险：造成明显维护性、扩展性、归属、事务、依赖或服务边界风险，后续很可能引发高成本缺陷。
- 中风险：局部设计、结构、重复、抽象、测试或协作问题，应尽快修复，但通常不单独阻塞发布。
- 建议优化：风险有限的合理改进；不要把局部清晰度、命名或小范围一致性问题夸大为高风险。

不确定时选择较低等级，除非长期影响具体、可解释且有证据支撑。

## Output Rules

初始审查输出必须包含：
1. 总体工程质量评估：架构质量、可维护性、可扩展性、技术债风险、模块边界健康度和协作或自维护准备度。
2. 风险摘要：各严重等级数量、合并/发布结论、阻塞问题和豁免状态。
3. 核心发现：每个问题包含位置、问题、原因、为什么重要、长期影响、建议方案和是否建议立即修改。
4. 需求/设计一致性审查。
5. 架构层建议。
6. 工程治理建议。
7. 相关时包含安全、测试和非功能质量审查。
8. 延后优化项：有价值但不紧急的事项。
9. 审查假设与缺口：缺失文档、未审查模块、测试不足或未验证运行时行为。
10. 默认修复优先级：至少给出“先修哪 1-3 个问题”“哪些可以晚点修”“哪些应先补测试/验证再改结构”。

普通 PR 或个人自审使用快速审查格式：
- 结论：`可以合并 / 暂缓合并 / 阻塞合并 / 可带债发布`
- 阻塞问题
- 主要风险
- 测试与验证
- 豁免/技术债
- 默认修复优先级
- 后续建议

架构较重或仓库级审查使用完整治理模板。

避免只写“建议优化”这类模糊结论；必须说明当前结构为什么不合理，以及更好的权衡是什么。

## Follow-Up Artifact Strategy

默认不生成大型后续文档。完成审查后，先提供可执行摘要，覆盖：
- 总体工程质量
- 核心风险
- 高优先级问题
- 默认修复优先级
- 架构健康度
- 是否存在需求/设计偏差
- 是否存在明显技术债

然后询问用户是否需要生成后续产物：

```text
是否需要我进一步生成以下内容？

1. 完整问题审查文档（review-report.md）
2. 修复计划文档（fix-plan.md）
3. 架构治理建议
4. 模块重构方案
5. 技术债治理路线
6. 分阶段整改计划
```

只创建用户选择的产物。每个产物都应限定在当前审查证据范围内，避免加入用户未要求的重流程内容。
